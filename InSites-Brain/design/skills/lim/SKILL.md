# LIM — Less-Is-More Review

**LIM means optimal, not minimal.** The goal is the right weight for the job — not the least text possible, not the most complete structure imaginable.

## The Principle

Every element in the reviewed output must earn its place by doing one of three things:

1. **Generating insight** — the reader/user learns something they didn't know
2. **Creating orientation** — the reader/user knows where they are, what happened, what's next
3. **Enabling dialogue** — the reader/user has enough to form an opinion, challenge, or extend

If an element does any of these — it stays, regardless of length.
If it doesn't — it goes, regardless of how "standard" or "complete" it feels.

## When to Cut

- **Ceremony without function**: boilerplate headings, formulaic closings, repeated context the reader already has, wrappers that add structure but not meaning
- **Scaffolding visible to the consumer**: internal organization that leaked into the output — checklists, process labels, meta-commentary about the output itself
- **Redundant decision points**: if a professional judgment can be made, don't ask — just do it. Eliminate choices that don't need to be choices.
- **Defensive verbosity**: restating things "just to be safe," adding caveats that protect the author but slow down the reader

## When NOT to Cut

- **Content that generates genuine thought** — questions, provocations, non-obvious observations. These are the core product.
- **Discoverable capabilities** — if the reader would benefit from knowing something exists, mention it. Don't hide value behind "on request" when they don't know they can request it.
- **Evidence and traceability** — citations, sources, references, reasoning chains. Weight that supports trust stays.
- **Core analysis** — dense work that produces real insight is not "verbose." It's the product.

## The Test

For every proposed change, ask:

> **"Does removing this make the reader's next decision easier, or does it just make the output shorter?"**

- If easier → cut.
- If just shorter → keep.

## How to Execute a LIM Review

When the user triggers LIM on a target (file, output, code, prompt, design):

### Step 1 — Read the target completely

Understand the full scope before making any judgment.

### Step 2 — Identify each element's job

For every section, function, paragraph, component, or block — what is it doing? Does it generate insight, create orientation, or enable dialogue?

### Step 3 — Classify

Create a table:

| Element | Earns its place? | Reason |
|---------|-------------------|--------|
| ... | ✅ Keep | Generates insight: [why] |
| ... | ✂️ Cut | Ceremony: [why] |
| ... | 🔄 Reshape | Good content, bad format: [how] |

### Step 4 — Propose changes

For each ✂️ and 🔄 item, show:
- **What changes** — concrete before/after or description
- **Why it qualifies as optimization** — not just reduction

### Step 5 — Summarize impact

| Metric | Before | After |
|--------|--------|-------|
| Elements reviewed | N | — |
| Kept as-is | N | — |
| Cut (ceremony/scaffolding) | N | — |
| Reshaped (better format) | N | — |
| Net effect | — | [one sentence] |

**Critical**: Never cut just to be shorter. LIM is about weight optimization, not weight loss.
